# Почему виджеты IntrinsicHeight и IntrinsicWidth считаются «дорогими»?

> Виджеты **IntrinsicHeight** и **IntrinsicWidth** считаются дорогими (expensive), потому что они нарушают базовое правило однопроходной компоновки Flutter (O(N) layout pass). Они заставляют движок выполнять предварительный **спекулятивный расчет (intrinsic sizing)** размеров своих дочерних элементов перед основным проходом layout. При наличии глубокой вложенности это приводит к экспоненциальному росту вычислений (O(N^2) и хуже), что вызывает пропуск кадров (UI jank).

## Разбор

### Протокол Layout во Flutter (Сложность O(N))

Архитектура Flutter построена вокруг жесткого правила: **Constraints go down. Sizes go up. Parent sets position.** (Ограничения спускаются вниз, размеры поднимаются вверх).
1. Родитель передает дочернему узлу жесткие ограничения (Constraints: minWidth, maxWidth, minHeight, maxHeight).
2. Ребенок выбирает свой размер строго в пределах этих ограничений.
3. Это позволяет Flutter обойти дерево Рендер-объектов ровно **один раз** (O(N)), гарантируя высокую производительность.

### Что ломают Intrinsic-виджеты?

Когда в коде появляется `IntrinsicWidth` (например, чтобы сделать все `ElevatedButton` в `Column` одинаковой ширины по самой широкой кнопке), однопроходный алгоритм ломается.

Flutter должен заранее узнать, какая кнопка самая широкая. Для этого запускается протокол вычисления внутренних размеров (Intrinsic Sizing Protocol):
1. `RenderIntrinsicWidth` вызывает у каждого своего ребенка методы: `computeMinIntrinsicWidth` или `computeMaxIntrinsicWidth`.
2. Дети начинают "спекулятивно" рассчитывать, сколько бы места они заняли, если бы у них не было жестких ограничений, запрашивая то же самое рекурсивно у своих детей.
3. Только получив максимальное/минимальное значение, `IntrinsicWidth` формирует строгие ограничения (Tight Constraints) и запускает **настоящий (второй)** проход компоновки (`performLayout`).

**Проблема вложенности (Экспоненциальный взрыв):**
Если один `Intrinsic`-виджет находится внутри другого (или внутри сложного поддерева, например таблицы), алгоритм начинает запрашивать размеры рекурсивно на каждом уровне. Это превращает временную сложность из линейной O(N) в полиномиальную O(N^2), O(N^3) и т.д. Дерево может быть пройдено сотни раз за один кадр.

*Пример проблемы (замедление интерфейса)*

```dart
// Этот код заставляет Row сделать высоту всех карточек 
// равной высоте самой высокой карточки с текстом.
IntrinsicHeight( // Первый проход (Intrinsic pass) + Второй проход (Layout pass)
  child: Row(
    crossAxisAlignment: CrossAxisAlignment.stretch,
    children: [
      Expanded(child: Text('Короткий текст')),
      // Тексту нужно рассчитать перенос строк, чтобы узнать свою естественную высоту
      Expanded(child: Text('Очень длинный текст, который\nразбивается\nна несколько строк')), 
    ],
  ),
)
```

### Кэширование Intrinsic значений

Чтобы смягчить удар по производительности, базовый класс `RenderBox` кэширует результаты вызовов `compute*Intrinsic*` внутри себя. Однако, любое малейшее изменение (анимация, смена шрифта, изменение текста или размера экрана) инвалидирует этот кэш, и тяжелые спекулятивные вычисления запускаются заново.

### Чем можно заменить?

* Использовать `CrossAxisAlignment.stretch` в связке с `Expanded` / `Flexible`, если размеры известны или определяются родителем.
* Заранее задавать фиксированные размеры (`SizedBox`).
* Использовать `CustomMultiChildLayout` или кастомный `RenderBox`, чтобы вручную реализовать сложную логику позиционирования, обходясь без спекулятивных проходов.
* Если `Intrinsic`-виджет жизненно необходим (иногда без него не обойтись при локализации и динамическом контенте), держите его как можно ниже (ближе к листьям) в дереве виджетов и категорически не допускайте вложенности таких виджетов друг в друга.
