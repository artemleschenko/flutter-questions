# Как работают Extension Methods под капотом? Можно ли добавить новое поле состояния внутрь класса через Extension?

> **Extension Methods** — это синтаксический сахар, реализующий механизм **статической диспетчеризации (Static Dispatch)**. Под капотом компилятор преобразует вызов метода расширения в вызов статической функции, куда целевой объект передается в качестве первого аргумента (this). Добавить новое поле состояния (переменную экземпляра) напрямую через расширение **нельзя**, так как расширения не могут изменить разметку объекта (memory layout) в оперативной памяти.

## Разбор

### Статическая диспетчеризация (Under the Hood)

В отличие от обычных методов класса, которые используют динамическую диспетчеризацию (поиск метода в таблице виртуальных методов - vtable во время выполнения), расширения разрешаются (resolve) **строго на этапе компиляции**.

*Пример кода*

```dart
extension StringCasingExtension on String {
  String toCapitalized() => length > 0 ? '${this[0].toUpperCase()}${substring(1).toLowerCase()}' : '';
}

void main() {
  String text = "hello";
  print(text.toCapitalized());
}
```

Для компилятора Dart класс `String` остается неизменным. Во время компиляции вызов `text.toCapitalized()` транслируется в статический вызов функции расширения:

```dart
// Псевдокод того, как это видит компилятор
StringCasingExtension_toCapitalized(text);
```

**Следствия статической диспетчеризации:**
1. **Не работают с `dynamic`:** Если переменная имеет тип `dynamic`, вызов метода расширения упадет в runtime (NoSuchMethodError), так как компилятор не знает тип объекта на этапе компиляции и не может подставить нужную статическую функцию.
   ```dart
   dynamic text = "hello";
   text.toCapitalized(); // NoSuchMethodError!
   ```
2. **Конфликты имен:** Если два расширения добавляют метод с одинаковым именем для одного типа, компилятор выдаст ошибку. Разрешить ее можно, явно указав имя расширения: `StringCasingExtension("hello").toCapitalized()` или скрыв конфликтное расширение при импорте (`import '...' hide OtherExtension;`).

### Почему нельзя добавить новое поле состояния?

Поскольку расширения — это просто набор внешних статических методов, они не могут изменить внутреннюю структуру самого класса.

Когда создается экземпляр класса, Dart VM (или AOT компилятор) выделяет строго определенный объем памяти под объект, зависящий от количества и типов его полей. Если бы расширение могло добавлять поля (например, `int myField;`), размер объекта в памяти пришлось бы динамически менять во время выполнения, что нарушило бы всю архитектуру ABI (Application Binary Interface), работу кэшей процессора и сборщика мусора (Garbage Collector).

*Единственный способ "эмулировать" состояние* в расширении — использовать внешний кэш (например, класс `Expando` или структуру `Map`), где ключом будет выступать сам объект.

*Пример эмуляции состояния (Expando)*

```dart
// Expando использует weak keys (слабые ссылки), поэтому не мешает сборщику 
// мусора очистить объект, если на него не осталось других сильных ссылок.
final Expando<int> _clickCounts = Expando<int>();

extension StatefulButton on Object {
  int get clickCount => _clickCounts[this] ?? 0;
  set clickCount(int value) => _clickCounts[this] = value;
}
```
