# Объясни концепцию «Во Flutter все есть виджет». Чем виджет отличается от обычного класса в Dart?

> Фраза **«во Flutter всё — виджет»** — это не буквальное правило для всего приложения, а описание **UI-слоя**: экран собирается из дерева **иммутабельных** объектов `Widget`, каждый из которых — «чертеж» части интерфейса. Виджет — обычный Dart-класс, но с контрактом фреймворка: он не рисует пиксели сам, а через `build()` описывает дочерние виджеты; фреймворк сопоставляет их с `Element` и `RenderObject` и обновляет экран через diff. От «обычного» класса виджет отличается тем, что он живёт внутри механики фреймворка (деревья/дифф/перестроения), его поля описывают **конфигурацию**, а UI собирается **композицией** (вложением виджетов), а не мутацией уже созданных view.

## Разбор

### Что на самом деле означает «всё — виджет»

Во Flutter пользовательский интерфейс целиком строится **из виджетов**: текст, отступы, жесты, тема, навигация — это классы, наследующие `Widget`. Отдельного XML или Storyboard нет: интерфейс описывается кодом, вкладывая виджеты друг в друга.

| Слой приложения | «Всё виджет»? | Примеры |
|-----------------|---------------|---------|
| UI (layout, стили, жесты) | Да | `Text`, `Padding`, `GestureDetector`, `Scaffold` |
| Состояние экрана | Частично | Долгоживущие объекты состояния — отдельные сущности, а `Widget` чаще выступает конфигурацией |
| Бизнес-логика, модели, API | Нет | `User`, `AuthRepository`, `http.Client` — обычные Dart-классы |

**Своими словами:** «всё — виджет» = «всё, что видит пользователь и как это расположено, выражается виджетами». Модели данных и сервисы остаются обычными классами — и это нормально.

### Виджет — не картинка на экране, а конфигурация

**Widget** — иммутабельное описание части UI. Создать `Column` с `Text` недостаточно, пока дерево не прикреплено к экрану через `runApp()`:

```dart
import 'package:flutter/material.dart';

void main() {
  runApp(
    const MaterialApp(
      home: Scaffold(
        body: Center(
          child: Text('Hello, World!'),
        ),
      ),
    ),
  );
}
```

`runApp` делает переданный виджет **корнем дерева**. До вызова `runApp` пользователь видит нативный splash; сам Flutter ещё не отрисовал UI.

В отличие от фреймворков на нативных view (`TextView`, `UILabel`) или WebView, Flutter рисует через **собственный движок** (Skia/Impeller): пиксели рисует фреймворк, а не платформа. Поэтому один и тот же виджет выглядит одинаково на разных ОС, а UI собирается **композицией** — один виджет вкладывается в другой, как дерево.

### Виджет vs обычный Dart-класс

| Критерий | Обычный класс | Виджет (`extends Widget`) |
|----------|---------------|---------------------------|
| Назначение | Любая логика / данные | Описание куска UI |
| Изменение полей | Можно мутировать (если не `final`) | Конфигурация через `final`, смена = новый виджет |
| Создание «детей» | Любым кодом в конструкторе/методах | Через `build()` → другие `Widget` |
| Жизненный цикл | GC, ваш код | Управляется Flutter через `Element` и сопоставление по `runtimeType`/`key` |
| Контекст дерева | Нет | `BuildContext` — доступ к данным «из дерева» (например, тема, размеры, локализация) |
| Отрисовка | Сами | `RenderObject` внутри фреймворка |

### Частые заблуждения

1. **«Виджет = элемент на экране»** — нет, это конфигурация; на экране работают `RenderObject`.
2. **«В приложении только виджеты»** — модели, репозитории, use case'ы — обычные классы.
3. **«Можно поменять поле виджета при тапе»** — антипаттерн; вместо мутации конфигурации пересобирают UI из новых данных (через обновление состояния).
4. **«Контекст — это просто “ссылка на виджет”»** — нет, `BuildContext` привязан к месту в дереве и нужен, чтобы доставать данные/настройки, предоставленные выше (тема, размеры, локализация).
