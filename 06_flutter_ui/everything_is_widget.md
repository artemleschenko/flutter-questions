# Объясни концепцию «Во Flutter все есть виджет». Чем виджет отличается от обычного класса в Dart?

> Фраза **«во Flutter всё — виджет»** — это не буквальное правило для всего приложения, а описание **UI-слоя**: экран собирается из дерева **иммутабельных** объектов `Widget`, каждый из которых — «чертеж» части интерфейса. Виджет — обычный Dart-класс, но с контрактом фреймворка: он не рисует пиксели сам, а через `build()` описывает дочерние виджеты; фреймворк сопоставляет их с `Element` и `RenderObject` и обновляет экран через diff. От «обычного» класса виджет отличается тем, что он живёт внутри механики фреймворка (деревья/дифф/перестроения), его поля описывают **конфигурацию**, а UI собирается **композицией** (вложением виджетов), а не мутацией уже созданных view.

## Разбор

### Что на самом деле означает «всё — виджет»

Во Flutter пользовательский интерфейс целиком строится **из виджетов**: текст, отступы, жесты, тема, навигация — это классы, наследующие `Widget`. Отдельного XML или Storyboard нет: интерфейс описывается кодом, вкладывая виджеты друг в друга.

| Слой приложения | «Всё виджет»? | Примеры |
|-----------------|---------------|---------|
| UI (layout, стили, жесты) | Да | `Text`, `Padding`, `GestureDetector`, `Scaffold` |
| Состояние экрана | Частично | Долгоживущие объекты состояния — отдельные сущности, а `Widget` чаще выступает конфигурацией |
| Бизнес-логика, модели, API | Нет | `User`, `AuthRepository`, `http.Client` — обычные Dart-классы |

**Своими словами:** «всё — виджет» = «всё, что видит пользователь и как это расположено, выражается виджетами». Модели данных и сервисы остаются обычными классами — и это нормально.

### Виджет — не картинка на экране, а конфигурация

**Widget** — иммутабельное описание части UI. Создать `Column` с `Text` недостаточно, пока дерево не прикреплено к экрану через `runApp()`:

```dart
import 'package:flutter/material.dart';

void main() {
  runApp(
    const MaterialApp(
      home: Scaffold(
        body: Center(
          child: Text('Hello, World!'),
        ),
      ),
    ),
  );
}
```

`runApp` делает переданный виджет **корнем дерева**. До вызова `runApp` пользователь видит нативный splash; сам Flutter ещё не отрисовал UI.

В отличие от фреймворков на нативных view (`TextView`, `UILabel`) или WebView, Flutter рисует через **собственный движок** (Skia/Impeller): пиксели рисует фреймворк, а не платформа. Поэтому один и тот же виджет выглядит одинаково на разных ОС, а UI собирается **композицией** — один виджет вкладывается в другой, как дерево.

### Декларативность: UI = f(state)

Когда состояние меняется, вы **не меняете** старый виджет — вы строите **новое описание**, а фреймворк сравнивает его с предыдущим и вносит минимальные правки в render-дерево:

```dart
// Нельзя: widget.color = Colors.red;
// Нужно: вернуть новый виджет с другим color
@override
Widget build(BuildContext context) {
  return Container(
    color: isError ? Colors.red : Colors.grey,
    child: Text(message),
  );
}
```

Это декларативный подход: `UI = f(state)`. Виджеты — аргументы этой функции; при смене состояния пересчитывается весь кусок UI, а не мутируются отдельные поля на экране.

### Три слоя: Widget → Element → RenderObject

Важно не смешивать **виджет** (лёгкий, пересоздаётся часто) и то, что реально рисуется:

| Слой | Роль | Жизненный цикл |
|------|------|----------------|
| **Widget** | Конфигурация: «как должно выглядеть» | Новый экземпляр при каждом `build` |
| **Element** | Связь виджета с местом в дереве, `BuildContext` | Переиспользуется, если тип и `key` совпали |
| **RenderObject** | Геометрия, layout, отрисовка | Обновляется diff'ом, дороже виджета |

```dart
class MyLabel extends StatelessWidget {
  const MyLabel({required this.text, super.key});

  final String text; // final — конфигурация неизменяема

  @override
  Widget build(BuildContext context) {
    return Text(text); // описание, не «рисование»
  }
}
```

**Обычный класс** `MyLabel` без `Widget` мог бы хранить `text` и метод `draw(Canvas c)` — вы сами управляли бы перерисовкой. **Виджет** делегирует это фреймворку: `build` возвращает описание, а `RenderObject` внизу дерева считает размеры и рисует.

### Что именно отличает виджет от «обычного» класса в Dart (развёрнуто)

С точки зрения языка Dart виджет — такой же класс. Отличие появляется из‑за **роли** этого класса в Flutter:

- **Жизненный цикл задаёт фреймворк, а не ваш код**. Экземпляры `Widget` создаются и выбрасываются постоянно: каждый `build()` возвращает новые объекты-конфигурации. При этом «привязка к месту на экране» — это не сам виджет, а `Element`, который фреймворк может переиспользовать между перестроениями (если совпали `runtimeType` и `key`).

- **Иммутабельность как контракт UI**. Виджет — это «снимок настроек» на текущий кадр: поля почти всегда `final`, и вместо «поменять поле» создаётся новый виджет с другими аргументами. Это делает UI предсказуемым: состояние хранится не в «view-объекте на экране», а в данных выше по дереву (или рядом), а UI просто пересчитывается из этих данных.

- **Композиция вместо мутации готовых view**. В императивных UI вы часто делаете так: «найди кнопку → поменяй текст → спрячь спиннер → добавь/удали view». Во Flutter подход другой: вы не «чините» уже созданные элементы, а описываете желаемый результат:

  - хотите отступ — добавляете `Padding(child: ...)`;
  - хотите обработать тап — оборачиваете в `GestureDetector(...)`;
  - хотите центрировать — оборачиваете в `Center(...)`.

  То есть поведение и оформление добавляются **обёртками**, а экран получается как функция композиции: `Screen = Center(Padding(GestureDetector(Text(...))))`.

- **Фреймворк сам решает, что реально обновить**. `build()` может возвращать «большое» новое дерево, но Flutter сравнит новую конфигурацию с предыдущей и применит минимум изменений внизу (в `RenderObject`). Поэтому «пересобрать UI» — не значит «перерисовать всё с нуля».

### Виджет vs обычный Dart-класс

| Критерий | Обычный класс | Виджет (`extends Widget`) |
|----------|---------------|---------------------------|
| Назначение | Любая логика / данные | Описание куска UI |
| Изменение полей | Можно мутировать (если не `final`) | Конфигурация через `final`, смена = новый виджет |
| Создание «детей» | Любым кодом в конструкторе/методах | Через `build()` → другие `Widget` |
| Жизненный цикл | GC, ваш код | Управляется Flutter через `Element` и сопоставление по `runtimeType`/`key` |
| Контекст дерева | Нет | `BuildContext` — доступ к данным «из дерева» (например, тема, размеры, локализация) |
| Отрисовка | Сами | `RenderObject` внутри фреймворка |

### Композиция вместо наследования

Сложный UI дробят на виджеты по ответственности: страница, кнопка, декоратор. Декорирующую обёртку (градиент, скругление) выносят в отдельный виджет — так `build` остаётся читаемым. Виджетам часто передают **другие виджеты** в конструктор (`title: Text('...')` в `AppBar`) — это гибче, чем глубокое наследование одного «супер-виджета».

### Частые заблуждения

1. **«Виджет = элемент на экране»** — нет, это конфигурация; на экране работают `RenderObject`.
2. **«В приложении только виджеты»** — модели, репозитории, use case'ы — обычные классы.
3. **«Можно поменять поле виджета при тапе»** — антипаттерн; вместо мутации конфигурации пересобирают UI из новых данных (через обновление состояния).
4. **«Контекст — это просто “ссылка на виджет”»** — нет, `BuildContext` привязан к месту в дереве и нужен, чтобы доставать данные/настройки, предоставленные выше (тема, размеры, локализация).
