Что такое Records в Dart 3 и как они позволяют избавиться от создания лишних классов-моделей?---------------------------------------------------------------------------------------

> TL;DR: `Records` в Dart 3 — это встроенные, анонимные и неизменяемые (immutable) агрегатные структуры данных, позволяющие группировать несколько значений различных типов в один объект. В отличие от обычных классов, они поддерживают структурную типизацию (structural typing) и обладают встроенным сравнением по значению (value equality), что позволяет возвращать из функций несколько значений и безопасно сравнивать их без создания DTO-классов или подключения пакета equatable.

### Разбор

**Структурная типизация против номинальной**

Классы в Dart используют номинальную типизацию: два класса с идентичными полями, но разными именами, считаются разными типами. `Records` не имеют имени. Их тип полностью определяется структурой — сигнатурой типов их полей.
Две записи `(int, String)` и `(int, String)`, созданные в совершенно разных участках кода, принадлежат к одному и тому же типу.

**Анатомия и синтаксис**

Записи могут содержать позиционные поля (доступ через `$`1, `$2` и т.д.) и именованные поля (доступ по имени).

**Встроенная эквивалентность**

Подобно `equatable`, записи автоматически переопределяют `==` и `hashCode`. Если структуры двух записей совпадают и значения всех их полей равны, сами записи равны. Это работает рекурсивно и глубоко (deep equality) для вложенных записей.

```dart
// Функция возвращает Record из позиционного поля (List) и именованного (bool)
(List<String>, {bool hasReachedMax}) fetchUsers() {
  return (['Alice', 'Bob'], hasReachedMax: false); // Анонимный возврат
}

void main() {
  final result = fetchUsers();
  
  // Доступ к полям
  print(result.$1); // ['Alice', 'Bob']
  print(result.hasReachedMax); // false

  // Деструктуризация (Pattern Matching)
  // Извлекаем значения сразу в новые локальные переменные
  final (users, hasReachedMax: isDone) = fetchUsers();

  // Структурное равенство в действии (true)
  final recordA = (1, name: 'Dart');
  final recordB = (1, name: 'Dart');
  print(identical(recordA, recordB)); // false (разные ссылки в памяти, если не const)
  print(recordA == recordB); // true (равенство по значению)
}
```

**Практическое применение**

В production-коде Records радикально очищают кодовую базу от "мусорных" классов-моделей.

Замена пакета `tuple` или `dartz`: Ранее для возврата пары значений (например,`Data` и `Error` при парсинге или запросе к API) приходилось использовать пакеты с `Tuple2`, `Tuple3` или `Either`. Теперь это решается нативно и строго типизировано.

BLoC / Riverpod: При передаче нескольких изолированных параметров в методы UI часто создавались промежуточные классы вроде `ProfileViewData`. Теперь метод-селектор или provider может просто вернуть запись (`userName`, `avatarUrl`), а виджет извлечет их через деструктуризацию.

Пагинация: Очень удобно возвращать из репозитория запись `(List<T> items, int nextPageKey)`.
