Что такое Tear-offs и как передача функции как объекта влияет на производительность и читаемость кода?---------------------------------------------------------------------------------------

> TL;DR: Tear-offs — это синтаксический механизм в Dart, позволяющий передавать ссылку на метод, функцию или конструктор как самостоятельный объект (замыкание) без их немедленного вызова (опуская скобки ()). Использование Tear-offs повышает читаемость, делая код более декларативным, и улучшает производительность за счет исключения аллокации памяти под промежуточные анонимные функции-обертки (лямбды).

### Разбор

**Механика под капотом (Создание замыкания)**

Когда вы пишете `obj.method` без круглых скобок, виртуальная машина Dart (или скомпилированный AOT-код) вычисляет получателя (`obj`), извлекает метод и неявно создает объект замыкания (closure). Это замыкание навсегда связывается с контекстом (конкретным экземпляром `obj`), сохраняя доступ к его состоянию (неявному `this`).

**Влияние на производительность (Memory Allocation & GC)**

Рассмотрим два варианта:

1. `() => obj.method()` — при каждом выполнении этого выражения Dart выделяет в куче (heap) память под новую анонимную функцию, которая внутри себя вызывает метод. Это создает дополнительный фрейм в стеке вызовов и увеличивает частоту работы Garbage Collector.
2. `obj.method` — Dart напрямую передает ссылку на метод, связанный с объектом. Более того, стандарт языка гарантирует кэширование/эквивалентность: извлечение одного и того же метода из одного и того же объекта всегда возвращает объекты, равные по значению (`==` вернет `true`).

**Tear-offs конструкторов (начиная с Dart 2.15)**

В современных версиях Dart появилась возможность делать Tear-off не только для методов, но и для конструкторов. Безымянный конструктор вызывается через `.new`, а именованный — через его имя (`.fromJson`).


```dart
class User {
  final int id;
  User(this.id);
  User.fromJson(Map<String, dynamic> json) : id = json['id'];
  
  void process() {}
}

void main() {
  final user = User(1);

  // 1. Сравнение Tear-offs (эквивалентность ссылок)
  final funcA = user.process;
  final funcB = user.process;
  print(funcA == funcB); // true! Dart оптимизирует это под капотом.
  
  final lambdaA = () => user.process();
  final lambdaB = () => user.process();
  print(lambdaA == lambdaB); // false. Каждый раз создается новый объект.

  // 2. Tear-offs конструкторов
  final ids = [1, 2, 3];
  
  // Плохо (лишняя лямбда):
  final users1 = ids.map((id) => User(id)).toList();
  
  // Хорошо (Tear-off конструктора через .new):
  final users2 = ids.map(User.new).toList(); 
}
```
